### Key decisions

**`bin` field**: Registers two executable aliases. `docker-compose-to-mermaid` is the canonical name; `dc2mermaid` is a short alias for convenience. Both point to the same compiled entry point.

**`files` field**: Controls what gets published to npm. Critically, `src/` is excluded — only the compiled `dist/` is shipped. This keeps the npm package small and avoids leaking development artifacts.

**`engines` field**: Node 18 is the minimum because it is the oldest LTS version still receiving security updates as of 2024. npm 9 is required because it ships with Node 18 and provides the `overrides` field behavior relied on by lock file resolution.

**`prepublishOnly`**: Runs a clean build, lint pass, and full test suite before any publish. This is a last-resort safeguard; the CI pipeline is the primary gate.

## 3. Branch Strategy

```
main          -- production, always releasable, protected
  └── develop -- integration branch, all feature PRs target here
        ├── feat/parse-networks
        ├── feat/add-cli-flags
        ├── fix/yaml-edge-case
        └── chore/update-deps
```

### Rules

| Branch | Direct push | Requires PR | Required checks | Who can merge |
|--------|-------------|-------------|-----------------|---------------|
| `main` | Never | Yes | CI, 1 review | Maintainers |
| `develop` | Never | Yes | CI | Any contributor |
| `feat/*`, `fix/*`, `chore/*` | Yes (author) | — | — | Author |

### Merge strategy

- `feature -> develop`: Squash merge. Keeps the develop log readable with one commit per feature.
- `develop -> main`: Merge commit. Preserves the history boundary so that release-please can compute the correct changelog range.

### Tagging

Tags are always cut on `main`. Never tag from a development branch.

## 5. CI Pipeline

**File**: `.github/workflows/ci.yml`

This pipeline runs on every pull request targeting `main` or `develop`, and on direct pushes to either branch. It validates the change is safe to merge: linting, type checking, unit tests, and a build verification.

```yaml
# .github/workflows/ci.yml

name: CI

on:
  pull_request:
    branches: [main, develop]
  push:
    branches: [main, develop]

# Cancel in-progress runs for the same branch to avoid
# wasting runner minutes on superseded commits.
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  # ------------------------------------------------------------------
  # Install and cache node_modules once.
  # All subsequent jobs restore from this cache rather than re-running
  # npm install, which makes the parallel jobs start immediately.
  # ------------------------------------------------------------------
  install:
    name: Install dependencies
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"

      - name: Install dependencies
        run: npm ci

      - name: Cache node_modules
        uses: actions/cache/save@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}

  # ------------------------------------------------------------------
  # Static analysis: ESLint + Prettier check.
  # Fails on any warning (--max-warnings 0) so warnings never
  # accumulate silently.
  # ------------------------------------------------------------------
  lint:
    name: Lint
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Restore node_modules
        uses: actions/cache/restore@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}
          fail-on-cache-miss: true

      - name: Lint
        run: npm run lint

      - name: Format check
        run: npm run format:check

  # ------------------------------------------------------------------
  # TypeScript compiler check (no emit).
  # Catches type errors that are not caught by tests.
  # ------------------------------------------------------------------
  typecheck:
    name: Type check
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Restore node_modules
        uses: actions/cache/restore@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}
          fail-on-cache-miss: true

      - name: Type check
        run: npm run typecheck

  # ------------------------------------------------------------------
  # Unit tests across the Node.js version matrix.
  # We test on Node 18 (minimum supported) and Node 20 (LTS).
  # The build artifact from the build job is used so we can also test
  # the compiled output, not just the TypeScript source.
  # ------------------------------------------------------------------
  test:
    name: Test (Node ${{ matrix.node-version }})
    needs: [install, build]
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false   # Run both matrix legs even if one fails
      matrix:
        node-version: ["18", "20"]
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}

      - name: Restore node_modules
        uses: actions/cache/restore@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}
          fail-on-cache-miss: true

      - name: Restore build artifact
        uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/

      - name: Run tests with coverage
        run: npm run test:coverage

      - name: Upload coverage report
        uses: actions/upload-artifact@v4
        if: matrix.node-version == '20'   # Only need one coverage report
        with:
          name: coverage
          path: coverage/
          retention-days: 7

  # ------------------------------------------------------------------
  # Build: compile TypeScript to dist/.
  # Stored as an artifact so test and other jobs reuse it rather than
  # each compiling independently.
  # ------------------------------------------------------------------
  build:
    name: Build
    needs: install
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Restore node_modules
        uses: actions/cache/restore@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}
          fail-on-cache-miss: true

      - name: Build
        run: npm run build:clean

      - name: Upload dist artifact
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 1

  # ------------------------------------------------------------------
  # Smoke test: install the package from the local tarball and run
  # the CLI against a fixture docker-compose.yml. This validates that
  # the package installs correctly and the CLI entry point works.
  # ------------------------------------------------------------------
  smoke-test:
    name: Smoke test (install from tarball)
    needs: build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Restore node_modules
        uses: actions/cache/restore@v4
        with:
          path: node_modules
          key: node-modules-${{ hashFiles('package-lock.json') }}
          fail-on-cache-miss: true

      - name: Restore dist
        uses: actions/download-artifact@v4
        with:
          name: dist
          path: dist/

      - name: Pack tarball
        run: npm pack --dry-run && npm pack

      - name: Install from local tarball
        run: |
          TARBALL=$(ls docker-compose-to-mermaid-*.tgz)
          npm install -g "./$TARBALL"

      - name: Run CLI smoke test
        run: |
          docker-compose-to-mermaid --version
          docker-compose-to-mermaid --input tests/fixtures/sample-compose.yml

  # ------------------------------------------------------------------
  # Final aggregation job for branch protection rules.
  # Branch protection requires this single job to pass rather than
  # enumerating every individual job. If any required job fails, this
  # job will not run and the PR is blocked.
  # ------------------------------------------------------------------
  ci-success:
    name: CI passed
    needs: [lint, typecheck, test, smoke-test]
    runs-on: ubuntu-latest
    if: always()
    steps:
      - name: Check all jobs passed
        run: |
          if [[ "${{ contains(needs.*.result, 'failure') }}" == "true" || \
                "${{ contains(needs.*.result, 'cancelled') }}" == "true" ]]; then
            echo "One or more required jobs failed"
            exit 1
          fi
          echo "All jobs passed"
```

### Job dependency graph

```
install
  ├── lint          (parallel)
  ├── typecheck     (parallel)
  └── build
        ├── test    (matrix: node 18, node 20)
        └── smoke-test
              └── ci-success  (aggregation)
```

Total wall-clock time on a typical run: approximately 3-4 minutes.

## 6. Release Pipeline

**File**: `.github/workflows/release.yml`

This pipeline uses [release-please](https://github.com/googleapis/release-please) to automate the release lifecycle. Release-please reads conventional commits since the last tag, bumps the version in `package.json`, updates `CHANGELOG.md`, and opens a release PR. When the maintainer merges that PR, this pipeline publishes to npm and creates a GitHub Release.

```yaml
# .github/workflows/release.yml

name: Release

on:
  push:
    branches: [main]

# Only one release can run at a time.
concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false   # Never cancel a release mid-flight

permissions:
  contents: write       # Create tags and GitHub releases
  pull-requests: write  # Open and update the release PR

jobs:
  # ------------------------------------------------------------------
  # release-please manages the version bump and changelog.
  # On every push to main it either:
  #   a) Creates/updates the "Release vX.Y.Z" PR if unreleased commits exist
  #   b) Creates a GitHub Release and tag when the release PR is merged
  # ------------------------------------------------------------------
  release-please:
    name: Release Please
    runs-on: ubuntu-latest
    outputs:
      # These outputs are only set when an actual release was created
      # (i.e., when the release PR was merged). Use them to gate the
      # publish and binary jobs.
      release_created: ${{ steps.release.outputs.release_created }}
      tag_name: ${{ steps.release.outputs.tag_name }}
      version: ${{ steps.release.outputs.version }}
      upload_url: ${{ steps.release.outputs.upload_url }}
    steps:
      - uses: googleapis/release-please-action@v4
        id: release
        with:
          release-type: node
          # release-please reads this to find the current version and
          # to know where to write the updated version on release.
          manifest-file: .release-please-manifest.json

  # ------------------------------------------------------------------
  # Publish to npm.
  # Only runs when release-please just created a release (i.e., the
  # release PR was merged and a tag was cut).
  # Uses NPM_TOKEN secret — set this in the repository secrets.
  # ------------------------------------------------------------------
  publish-npm:
    name: Publish to npm
    needs: release-please
    if: needs.release-please.outputs.release_created == 'true'
    runs-on: ubuntu-latest
    environment: npm-publish   # Requires manual approval for production publishes
    steps:
      - uses: actions/checkout@v4
        with:
          # Check out the tag that was just created, not HEAD of main,
          # to ensure we publish exactly what was tagged.
          ref: ${{ needs.release-please.outputs.tag_name }}

      - name: Setup Node.js with npm registry auth
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          registry-url: "https://registry.npmjs.org"

      - name: Install dependencies
        run: npm ci

      - name: Build
        run: npm run build:clean

      - name: Run tests (final gate before publish)
        run: npm test

      - name: Publish to npm
        run: npm publish --access public
        env:
          # NPM_TOKEN must be a "Granular Access Token" scoped to the
          # specific package with publish permission. Do not use a
          # legacy token or automation token with broad scope.
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}

      - name: Verify publish
        run: |
          # Wait for npm CDN propagation (typically 30-60 seconds)
          sleep 60
          npm view docker-compose-to-mermaid@${{ needs.release-please.outputs.version }} version

  # ------------------------------------------------------------------
  # Build and upload standalone binaries.
  # Runs in parallel with npm publish after a release is created.
  # Each binary is uploaded to the GitHub Release assets.
  # ------------------------------------------------------------------
  build-binaries:
    name: Build binary (${{ matrix.target }})
    needs: release-please
    if: needs.release-please.outputs.release_created == 'true'
    runs-on: ubuntu-latest
    strategy:
      matrix:
        include:
          - target: node18-linux-x64
            artifact: docker-compose-to-mermaid-linux-x64
          - target: node18-macos-x64
            artifact: docker-compose-to-mermaid-macos-x64
          - target: node18-win-x64
            artifact: docker-compose-to-mermaid-win-x64.exe
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ needs.release-please.outputs.tag_name }}

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: "20"

      - name: Install dependencies
        run: npm ci

      - name: Build TypeScript
        run: npm run build:clean

      - name: Package binary
        run: |
          npx pkg dist/cli.js \
            --target ${{ matrix.target }} \
            --output binaries/${{ matrix.artifact }}

      - name: Upload binary to GitHub Release
        uses: actions/upload-release-asset@v1
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          upload_url: ${{ needs.release-please.outputs.upload_url }}
          asset_path: binaries/${{ matrix.artifact }}
          asset_name: ${{ matrix.artifact }}
          asset_content_type: application/octet-stream

  # ------------------------------------------------------------------
  # Update the Homebrew tap formula after a successful npm publish.
  # This dispatches a workflow in the homebrew-tap repository.
  # ------------------------------------------------------------------
  update-homebrew-tap:
    name: Update Homebrew tap
    needs: [release-please, publish-npm]
    if: needs.release-please.outputs.release_created == 'true'
    runs-on: ubuntu-latest
    steps:
      - name: Trigger Homebrew tap update
        uses: actions/github-script@v7
        with:
          # TAP_REPO_TOKEN must have `repo` scope on the homebrew-tap
          # repository.
          github-token: ${{ secrets.TAP_REPO_TOKEN }}
          script: |
            await github.rest.actions.createWorkflowDispatch({
              owner: context.repo.owner,
              repo: 'homebrew-tap',
              workflow_id: 'update-formula.yml',
              ref: 'main',
              inputs: {
                package: 'docker-compose-to-mermaid',
                version: '${{ needs.release-please.outputs.version }}'
              }
            });
```

### Required secrets

| Secret | Purpose | Scope |
|--------|---------|-------|
| `NPM_TOKEN` | Publish to npm | Granular token: this package, publish |
| `TAP_REPO_TOKEN` | Trigger Homebrew tap workflow | `repo` on homebrew-tap repo |
| `GITHUB_TOKEN` | Built-in, create releases, upload assets | Automatic |

### Release environment protection

Configure a GitHub environment named `npm-publish` with:
- Required reviewers: 1 maintainer
- Deployment branches: `main` only

This prevents accidental or malicious npm publishes even if the workflow is triggered unexpectedly.

## 7. Semantic Versioning Strategy

### Tooling: release-please

release-please is preferred over semantic-release for this project because:
1. It creates an explicit PR for each release, giving maintainers a review point before anything is published.
2. It does not require a `GITHUB_TOKEN` with elevated permissions at commit time.
3. The PR model makes version bumps auditable and reversible before they hit npm.

### .release-please-manifest.json

```json
{
  ".": "0.1.0"
}
```

### release-please-config.json

```json
{
  "packages": {
    ".": {
      "release-type": "node",
      "changelog-sections": [
        { "type": "feat",     "section": "Features" },
        { "type": "fix",      "section": "Bug Fixes" },
        { "type": "perf",     "section": "Performance Improvements" },
        { "type": "revert",   "section": "Reverts" },
        { "type": "docs",     "section": "Documentation", "hidden": false },
        { "type": "refactor", "section": "Code Refactoring", "hidden": true },
        { "type": "test",     "section": "Tests", "hidden": true },
        { "type": "build",    "section": "Build System", "hidden": true },
        { "type": "ci",       "section": "CI", "hidden": true }
      ]
    }
  }
}
```

### Bump rules

| Commit prefix | Version bump | Example |
|---------------|-------------|---------|
| `feat:` | Minor (0.X.0) | `feat: add network diagram support` |
| `fix:` | Patch (0.0.X) | `fix: handle missing image field` |
| `feat!:` or `BREAKING CHANGE:` | Major (X.0.0) | `feat!: rename --output flag to --out` |
| `perf:`, `refactor:`, `docs:` | Patch | `docs: update README` |

### Pre-release versions

For `develop` → `main` candidate testing:
```bash
# Publish a pre-release manually from develop
npm version prerelease --preid=beta   # 1.2.0-beta.0
npm publish --tag beta
```

Users can install pre-releases with: `npm install -g docker-compose-to-mermaid@beta`

## 8. GitHub Action Definition

This is the action that users drop into their own workflows. The action runs the CLI tool against a target docker-compose file and commits (or outputs) the resulting Mermaid diagram.

**File**: `action.yml` (at repository root)

```yaml
# action.yml
# This file defines the GitHub Action that users consume.
# Users reference it as: uses: YOUR_ORG/docker-compose-to-mermaid@v1

name: "Docker Compose to Mermaid"
description: "Generate a Mermaid architecture diagram from a Docker Compose file"
author: "YOUR_ORG"

branding:
  icon: "box"
  color: "blue"

inputs:
  input:
    description: "Path to the docker-compose file to parse"
    required: false
    default: "docker-compose.yml"

  output:
    description: "Path to write the generated Mermaid diagram"
    required: false
    default: "docs/architecture.md"

  commit:
    description: "Whether to commit the generated diagram back to the repository"
    required: false
    default: "true"

  commit-message:
    description: "Commit message for the auto-generated diagram commit"
    required: false
    default: "docs: update architecture diagram [skip ci]"

  fail-on-change:
    description: |
      Fail the workflow if the diagram changed (useful for enforcing
      that diagrams are kept up to date in PRs).
    required: false
    default: "false"

  diagram-type:
    description: "Type of Mermaid diagram to generate (architecture, flowchart, c4)"
    required: false
    default: "architecture"

outputs:
  diagram:
    description: "The generated Mermaid diagram content"
    value: ${{ steps.generate.outputs.diagram }}

  changed:
    description: "Whether the diagram changed compared to the previous version"
    value: ${{ steps.check-diff.outputs.changed }}

runs:
  using: "composite"
  steps:
    - name: Setup Node.js
      uses: actions/setup-node@v4
      with:
        node-version: "20"

    - name: Install docker-compose-to-mermaid
      shell: bash
      # Pin to the exact version matching the action tag so the tool
      # version and action version stay in sync.
      run: npm install -g docker-compose-to-mermaid@${{ github.action_ref }}

    - name: Generate diagram
      id: generate
      shell: bash
      run: |
        OUTPUT=$(docker-compose-to-mermaid \
          --input "${{ inputs.input }}" \
          --type "${{ inputs.diagram-type }}")

        # Write to the output file
        mkdir -p "$(dirname '${{ inputs.output }}')"
        echo "$OUTPUT" > "${{ inputs.output }}"

        # Expose diagram content as step output (truncated for very large diagrams)
        echo "diagram<<EOF" >> "$GITHUB_OUTPUT"
        echo "$OUTPUT" >> "$GITHUB_OUTPUT"
        echo "EOF" >> "$GITHUB_OUTPUT"

    - name: Check for changes
      id: check-diff
      shell: bash
      run: |
        if git diff --quiet "${{ inputs.output }}"; then
          echo "changed=false" >> "$GITHUB_OUTPUT"
        else
          echo "changed=true" >> "$GITHUB_OUTPUT"
        fi

    - name: Fail if diagram changed and fail-on-change is set
      shell: bash
      if: inputs.fail-on-change == 'true' && steps.check-diff.outputs.changed == 'true'
      run: |
        echo "Architecture diagram is out of date."
        echo "Run docker-compose-to-mermaid locally and commit the result."
        git diff "${{ inputs.output }}"
        exit 1

    - name: Commit updated diagram
      shell: bash
      if: inputs.commit == 'true' && steps.check-diff.outputs.changed == 'true'
      run: |
        git config user.name "github-actions[bot]"
        git config user.email "github-actions[bot]@users.noreply.github.com"
        git add "${{ inputs.output }}"
        git commit -m "${{ inputs.commit-message }}"
        git push
```

### Why composite action (not Docker)?

A composite action (shell steps) is chosen over a Docker container action for this tool because:
1. Startup time: Docker container actions pull an image on every run (+30-60 seconds). Composite actions start immediately.
2. The tool is pure Node.js with no native dependencies — there is no need for a controlled Linux environment.
3. Composite actions work on all runner OS types (ubuntu, macos, windows), which is important for developers who use macOS-hosted runners or self-hosted runners.

A Docker container action would be appropriate if the tool needed specific system libraries or needed to guarantee a precise OS environment.

### Versioning the action

Users pin to a major version tag (`@v1`) rather than a full semver tag. Maintain floating major version tags:

```bash
# After releasing v1.2.0, also move the v1 tag forward:
git tag -fa v1 -m "Update v1 to v1.2.0"
git push origin v1 --force
```

This is the standard GitHub Actions pattern: users who pin `@v1` automatically get patch and minor updates, but are protected from breaking changes which get a new major tag.

## 9. Example User Workflow

This is the workflow file users drop into their own repositories to get automatic diagram generation.

```yaml
# .github/workflows/update-architecture-diagram.yml
# Add this to your repository to automatically keep your architecture
# diagram in sync with your docker-compose.yml.

name: Update Architecture Diagram

on:
  push:
    branches: [main]
    paths:
      # Only run when docker-compose files actually change.
      # This avoids unnecessary runs on unrelated commits.
      - "docker-compose.yml"
      - "docker-compose.override.yml"
      - "docker-compose.*.yml"
      - "compose.yml"
      - "compose.yaml"

  pull_request:
    paths:
      - "docker-compose.yml"
      - "docker-compose.override.yml"
      - "docker-compose.*.yml"

  # Allow manual trigger for initial diagram generation
  workflow_dispatch:

permissions:
  contents: write   # Required to commit the diagram back to the repo

jobs:
  update-diagram:
    name: Update Mermaid diagram
    runs-on: ubuntu-latest
    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          # Use a token with write access if committing back.
          # For public repos, GITHUB_TOKEN is sufficient.
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Generate architecture diagram
        uses: YOUR_ORG/docker-compose-to-mermaid@v1
        with:
          input: docker-compose.yml
          output: docs/architecture.md
          commit: ${{ github.event_name == 'push' }}
          # On PRs, don't commit — just check if the diagram is current
          fail-on-change: ${{ github.event_name == 'pull_request' }}
          commit-message: "docs: update architecture diagram [skip ci]"
```

### Behaviour per trigger

| Trigger | `commit` | `fail-on-change` | Result |
|---------|----------|-----------------|--------|
| `push` to `main` | `true` | `false` | Diagram is committed automatically after merge |
| `pull_request` | `false` | `true` | PR fails if diagram is not already up to date |
| `workflow_dispatch` | `true` | `false` | Diagram is regenerated and committed on demand |

The `[skip ci]` in the commit message prevents the diagram commit from triggering a recursive workflow run.

## 10. Standalone Binary Packaging

Standalone binaries allow users to run the tool without Node.js installed, which is valuable for DevOps environments where installing Node is undesirable.

### Tool: pkg

[`pkg`](https://github.com/vercel/pkg) bundles the Node.js runtime with the application into a single executable.

### package.json pkg configuration

```json
{
  "pkg": {
    "scripts": "dist/**/*.js",
    "assets": [],
    "targets": [
      "node18-linux-x64",
      "node18-macos-x64",
      "node18-win-x64"
    ],
    "outputPath": "binaries/"
  }
}
```

### Build script

```bash
npm run build:clean
npx pkg . --compress GZip
```

The `--compress GZip` flag reduces binary size by roughly 30-40%.

### Binary naming convention

```
docker-compose-to-mermaid-linux-x64          (Linux x86_64)
docker-compose-to-mermaid-macos-x64          (macOS Intel)
docker-compose-to-mermaid-macos-arm64        (macOS Apple Silicon — future)
docker-compose-to-mermaid-win-x64.exe        (Windows x86_64)
```

### Distribution

Binaries are attached to GitHub Releases as assets (handled by the release pipeline in section 6). Users can download them from the GitHub Releases page or via direct URL:

```bash
# Linux install without npm
curl -L https://github.com/YOUR_ORG/docker-compose-to-mermaid/releases/latest/download/docker-compose-to-mermaid-linux-x64 \
  -o /usr/local/bin/docker-compose-to-mermaid
chmod +x /usr/local/bin/docker-compose-to-mermaid
```

### macOS Apple Silicon note

`pkg` does not natively support `node18-macos-arm64`. Options:
1. Build the arm64 binary on a macOS Apple Silicon runner (`runs-on: macos-latest` now defaults to M1).
2. Use `node18-macos-x64` with Rosetta 2 (runs transparently on Apple Silicon).

Option 1 is recommended for production once the runner is available. Add a dedicated matrix entry when ready.

## 11. Homebrew Tap Strategy

A Homebrew tap allows macOS users to install with `brew install YOUR_ORG/tap/docker-compose-to-mermaid` without Node.js.

### Repository structure

Create a separate repository: `YOUR_ORG/homebrew-tap`

```
homebrew-tap/
├── Formula/
│   └── docker-compose-to-mermaid.rb
└── .github/
    └── workflows/
        └── update-formula.yml
```

### Formula template

```ruby
# Formula/docker-compose-to-mermaid.rb
class DockerComposeToMermaid < Formula
  desc "Convert Docker Compose configurations into Mermaid architecture diagrams"
  homepage "https://github.com/YOUR_ORG/docker-compose-to-mermaid"
  url "https://github.com/YOUR_ORG/docker-compose-to-mermaid/releases/download/v__VERSION__/docker-compose-to-mermaid-macos-x64"
  sha256 "__SHA256__"
  version "__VERSION__"
  license "MIT"

  # No dependencies — standalone binary bundles Node.js runtime

  def install
    bin.install "docker-compose-to-mermaid-macos-x64" => "docker-compose-to-mermaid"
    # Optionally install the short alias
    bin.install_symlink "docker-compose-to-mermaid" => "dc2mermaid"
  end

  test do
    system "#{bin}/docker-compose-to-mermaid", "--version"
  end
end
```

### Auto-update workflow (in homebrew-tap repo)

```yaml
# .github/workflows/update-formula.yml
name: Update formula

on:
  workflow_dispatch:
    inputs:
      package:
        required: true
      version:
        required: true

jobs:
  update:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Download macOS binary and compute SHA256
        run: |
          VERSION="${{ github.event.inputs.version }}"
          URL="https://github.com/YOUR_ORG/docker-compose-to-mermaid/releases/download/v${VERSION}/docker-compose-to-mermaid-macos-x64"
          curl -L "$URL" -o binary
          SHA256=$(sha256sum binary | awk '{print $1}')
          echo "SHA256=$SHA256" >> "$GITHUB_ENV"
          echo "VERSION=$VERSION" >> "$GITHUB_ENV"

      - name: Update formula
        run: |
          sed -i "s/__VERSION__/$VERSION/g" Formula/docker-compose-to-mermaid.rb
          sed -i "s/__SHA256__/$SHA256/g" Formula/docker-compose-to-mermaid.rb
          # Replace version in url line as well
          sed -i "s|/download/v[^/]*/|/download/v$VERSION/|g" Formula/docker-compose-to-mermaid.rb

      - name: Commit and push
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add Formula/docker-compose-to-mermaid.rb
          git commit -m "chore: update docker-compose-to-mermaid to v${{ env.VERSION }}"
          git push
```

### User install commands

```bash
# First time
brew tap YOUR_ORG/tap
brew install docker-compose-to-mermaid

# Update
brew upgrade docker-compose-to-mermaid
```

Same as `main`, except:

### Tag protection

Protect `v*` tags:
- GitHub → Settings → Tags → Add rule: `v*`

GitHub → Settings → Actions → General:
- Allow actions and reusable workflows
- Allow actions created by GitHub
- Allow actions from Marketplace (only trusted publishers)
- Disable: Fork pull request workflows from outside collaborators (require approval)

This last setting prevents untrusted contributors from running workflows that could exfiltrate secrets.

Create two environments:

**`npm-publish`**:
- Required reviewers: 1 (lead maintainer)

**`github-release`** (optional, for additional gate on release asset uploads):

## 13. Security Considerations

### Secret management

- `NPM_TOKEN`: Use a Granular Access Token scoped to exactly this package with `publish` permission only. Rotate annually or on team member departure.
- `TAP_REPO_TOKEN`: Fine-grained personal access token with `contents: write` on the homebrew-tap repository only.
- Never use `GITHUB_TOKEN` as a substitute for `NPM_TOKEN` — they serve different registries.

### Supply chain

- Pin all GitHub Actions to their full SHA in production-critical workflows:
  ```yaml
  # Instead of:
  uses: actions/checkout@v4
  # Use:
  uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
  ```
  This protects against a compromised action tag being moved to malicious code.

- Run `npm audit` in CI. Add to the lint job:
  ```yaml
  - name: Audit dependencies
    run: npm audit --audit-level=high
  ```

- Use `npm ci` everywhere in CI (never `npm install`). `npm ci` is deterministic and fails on lock file mismatches.

### npm publish security

- Enable 2FA on the npm account used to publish.
- Enable `npm publish` 2FA enforcement at the package level: `npm access set mfa=publish docker-compose-to-mermaid`
- After publish, always verify the package contents: `npm pack --dry-run` shows exactly what was included.

### Permissions in workflows

Follow least privilege. Every workflow that doesn't need a permission should explicitly set it to `none` or omit it (default is read-only for most):

```yaml
permissions:
  contents: read    # Default: just checkout
  # Add only what the job specifically needs
```

The release workflow explicitly needs `contents: write` and `pull-requests: write` for release-please.

### Dependabot configuration

Create `.github/dependabot.yml` to keep dependencies current:

```yaml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    # Group minor and patch updates to reduce PR noise
    groups:
      minor-and-patch:
        update-types: ["minor", "patch"]
    # Open PRs for major bumps individually so they get reviewed carefully
    open-pull-requests-limit: 5

  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "monthly"
    groups:
      actions:
        patterns: ["*"]
```
